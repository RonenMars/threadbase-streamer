### 1.2 WebSocket — `/ws`

**Inbound messages (client → server):**

| `type` | Description |
|--------|-------------|
| `register` | Associates `clientId` with the WebSocket connection |
| `subscribe_session` | Request terminal replay + future output for a session |
| `hold_session` | Immediately trigger PTY grace-kill for a session |

**Outbound events (server → client):**

| `type` | Trigger |
|--------|---------|
| `session_list` | On WS connect; on any session start/resume/adopt/cancel |
| `cache_ready` | When SQLite cache finishes warm-up |
| `scan_progress` | During initial directory scan (`{scanned, total}`) |
| `session_ready` | PTY reached first prompt (`waiting_input`) |
| `session_update` | PTY status change (running → waiting_input → idle) |
| `terminal_output` | Raw PTY output chunk |
| `terminal_replay` | On `subscribe_session` — last 200 rendered screen lines |
| `conversation_event` | Single JSONL line appended (legacy compat) |
| `conversation_events` | Batch of JSONL lines appended (newer clients) |
| `agent_output` | Agent turn content from progress webhook |
| `stage_transition` | Agent stage change from progress webhook |
| `turn_failure` | Agent turn failed from progress webhook |

### 1.3 Major Async Flows

**PTY session lifecycle:**
1. `POST /api/sessions/start` → `PTYManager.startFresh()` → spawns `claude` subprocess → `pendingReady` gating → `markReady()` on prompt marker → `session_ready` WS event → user input via `sendInput()` → `pty.input_write` → `pty.input_submit` → `terminal_output` WS broadcast → PTY exit → `session_update(idle)`.
2. `POST /api/sessions/resume` → `PTYManager.start(sessionId, --resume)` → same prompt-marker path.

**WS subscriber grace timer:**
- Last subscriber disconnects → `startGraceTimer(sessionId, ptyGracePeriodMs=270s)` → `PTYManager.putOnHold()` → PTY SIGINT → `session_update(idle)`.
- `hold_session` message → `startGraceTimer(sessionId, 0)` (instant).

**Conversation watcher (JSONL tail):**
- `ConversationWatcher.watch(filePath)` via chokidar → `readNewLines()` on `change`/`add` → `onNewLines` callback → `cache.updateFromLines()` + WS `conversation_events` broadcast.
- `watchDirectory(dir)` → `onConversationChanged` → `cache.invalidateByFilePath()` + debounced scanner-staleness flip.

**Cache warm-up (startup):**
- `ConversationScanner` discovers all JSONLs → `cache.upsertFull()` for each → `cache_ready` WS broadcast. Runs on first `GET /api/conversations` or `GET /api/search` that needs fresh data.

**Auto-update:**
- `POST /api/__update` (HMAC webhook) or `tb-streamer update` CLI → `checkForUpdate()` → `download()` → `unpack()` → `install()` → `swap()` → `restart()`.

**Prod/dev lifecycle:**
- Dev server writes `~/.threadbase/prod-suspended.json` marker → `launchd-entry.ts` reads it on crash to decide whether to auto-restore prod.

### 3.3 State transitions with no observability

- **PTY process exit**: `handleExit()` in `pty-manager.ts` sets `status=idle`, records `failureReason` on fast-fail, calls `onStatusChange` — but emits **no log line**. Diagnosing why a session went idle requires cross-referencing the WS `session_update` event with PTY chunk logs.
- **WS connect / disconnect**: No log when a client opens or closes the `/ws` endpoint. `wsHub.addClient()` and the `close` event handler are silent. Impossible to know how many simultaneous mobile clients are connected.
- **`subscribe_session` received**: No log. Can't tell when/whether mobile subscribed to a session.
- **Pair token consumed / expired / rejected**: `pairTokens.consume()` returns `{ok: false, reason}` — the 401 response is sent but not logged.
- **Pair token rate-limit hit**: 429 returned with no log.
- **`hold_session` message received**: `startGraceTimer(sessionId, 0)` called with no log.
- **Auth middleware bypass (localNoAuth)**: Local request bypasses auth silently — no `debug` log.
- **Cache-tail fallback path**: `handleGetConversation` serves cached tail when JSONL is gone — no log. Hard to diagnose missing-file fallbacks in prod.
- **Agent progress dedupe**: `deduped: true` returned with no log.
- **Agent progress HMAC failure**: Returns 401 with no log.
- **`enrichResumedSessionAsync` errors**: Uses raw `console.error` (`server.ts:1720`) instead of the structured logger — not captured by pino.

## 4. Request Trace Opportunities

### 4.1 Request logs

**Status: already done.** `api/app.ts:38-48` logs every HTTP request with method, path, status, and ms. The only gap is that `ALREADY_HANDLED` sentinel responses (status 597) log as `597` in the middleware — the true HTTP status from `json(res, N, ...)` is written to the `ServerResponse` directly, not visible to Hono's middleware. The log shows 597 for all session/conversation routes.

**Fix location**: `api/app.ts:38` — resolve 597 → actual status or move timing into the handler layer. Alternatively, add a custom response header (e.g. `X-Tb-Status`) inside `json()` in `server.ts` that the middleware reads back.

### 4.2 Access logs (auth)

**No coverage.** Add logging at two points:

```
src/api/middleware/auth.middleware.ts — after each decision branch:
  - Local bypass:   log.debug("[auth] local bypass", { path, remoteAddr })
  - Bearer pass:    log.debug("[auth] bearer ok", { path })
  - Key param pass: log.debug("[auth] key-param ok", { path })
  - 401:            log.warn("[auth] rejected", { path, method, remoteAddr })
```

Rate-limit hit on pair exchange should also log (`server.ts` around the `checkExchangeRateLimit` call before the 429 response).

### 4.3 Latency logs

Specific operations that are slow in production but have zero timing instrumentation:

| Operation | File:function | What to measure |
|-----------|--------------|-----------------|
| SQLite `list()` (conversation list) | `conversation-cache.ts#list` | Full scan time; grows with corpus |
| SQLite `upsertFull()` (cache warm-up) | `conversation-cache.ts#upsertFull` | Per-file write time during startup |
| `findConversationByUuid()` | `server.ts#findConversationByUuid` | Scanner lookup + optional refresh |
| `ptyManager.getOutputLines()` | `pty-manager.ts#getOutputLines` | xterm flush + scroll-buffer read; can be >50ms on large sessions |
| `readCwdFromJsonl()` | `server.ts#readCwdFromJsonl` | File read for JSONL cwd; called on every resume |
| Scanner warm-up sweep | `server.ts` around `scanner.scan()` | Already has `scan_progress` WS events but no pino timing |
| `checkForUpdate()` + `fetchLatestRelease()` | `updater/check-update.ts`, `updater/github-releases.ts` | Network round-trip to GitHub API |
| `download()` tarball | `updater/download.ts` | Download duration + bytes |

**Suggested pattern** (example for `getOutputLines`):
```typescript
const t0 = Date.now();
const lines = await this.ptyManager.getOutputLines(sessionId, 200);
this.log.debug("[replay] getOutputLines", { sessionId, lines: lines.length, ms: Date.now() - t0 });
```

## 5. Prioritized Recommendations

**Rank 1 — Auth 401 logging** (`src/api/middleware/auth.middleware.ts`)
Add `log.warn("[auth] 401", {path, method, remoteAddr})` on the final `return c.json({error:"Unauthorized"},401)`. This is a security signal — silent 401s make it impossible to detect brute-force or mis-configured clients. One line.

**Rank 2 — PTY exit log** (`src/pty-manager.ts#handleExit`)
Add `this.log.info("[pty.exit] ...")` with `{sessionId, exitCode, elapsedMs, failureReason}`. Currently a session going idle leaves no trace unless you notice the `session_update` WS event.

**Rank 3 — WS connect/disconnect + client count** (`src/ws-hub.ts#addClient` and the `close` callback)
Two `log.info` calls give a full picture of concurrent mobile load. Essential for diagnosing "why did the PTY die?" (no subscribers → grace timer fired).

**Rank 4 — Fix 597 status in request log** (`src/api/app.ts:38`)
The HTTP request middleware logs status 597 for every session/conversation route. Either read an `X-Tb-Status` response header set by `json()`, or rewrite the ALREADY_HANDLED routes to use native Hono responses. Without this, the access log is largely useless for those endpoints.

**Rank 5 — Error middleware logs** (`src/api/middleware/error.middleware.ts`)
Add `log.error("[http] unhandled error", {path, method, err})` before the 500 response. Currently any uncaught exception in a Hono handler is silently swallowed into a `{error: "..."}` body.

**Rank 6 — Conversation watcher errors** (`server.ts` `ConversationWatcher` constructor call, ~line 208)
Wire the missing `onError` callback. The `ConversationWatcher` supports it (`conversationWatcher.ts:20`) but `server.ts` never passes it. File-tail errors (permission denied, deleted file mid-read) are silently lost.

**Rank 7 — Updater structured logging** (`src/updater/check-update.ts`, `download.ts`, `install.ts`, `swap.ts`, `restart.ts`)
The entire update pipeline has zero pino logs. Add at minimum: check result, download start/end (with bytes), install complete, swap complete, restart attempt. These five files collectively handle a critical operation with no audit trail.

**Rank 8 — `subscribe_session` log** (`server.ts:394`)
One `log.debug("[ws] subscribe_session", {sessionId, clientId})` reveals which mobile client subscribed to which session. Critical for debugging "mobile sees empty screen" issues.

**Rank 9 — Slow-query latency on `handleListConversations`** (`server.ts#handleListConversations` ~line 922)
Wrap the `cache.list()` call with `Date.now()` before/after. This query touches every cached row and grows linearly — it's the most likely hot path to degrade first.

**Rank 10 — Ring-buffer pressure event** (`pty-manager.ts#handleOutput`, ~line 565)
When the output buffer is trimmed (i.e. `outputBuffer.length > OUTPUT_BUFFER_MAX`), emit a `pty.buffer_trimmed` log with `droppedBytes`. Currently silent — the only way to know output was lost is noticing replayed sessions look truncated.
