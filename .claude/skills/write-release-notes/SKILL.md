# write-release-notes

## Overview

`tb-streamer` is a Hono HTTP/WebSocket streaming server for Claude Code sessions. It ships in two layers of changelogs:

- **`CHANGELOG.md`** at the repo root — auto-generated by semantic-release, one line per conventional commit. Granular, machine-friendly, regenerated on every release tag. **Do not hand-edit.**
- **`docs/release-notes/YYYY-MM-DD-<milestone>.md`** — the human narrative of what shipped in a milestone, why it matters, what's deferred, breaking changes, operator-facing additions. Written once per milestone, by hand (or by this skill), and lives in git as the durable story.

This skill drafts the second layer. It produces a single markdown file under `docs/release-notes/` based on the structure of `docs/release-notes/_template.md`, scoped to the commits and PRs that landed since the last release-notes file.

The two layers coexist deliberately. `CHANGELOG.md` answers "what commits shipped in v1.5.2?". This directory answers "what was Milestone B actually about, and what should an operator or mobile-client author know before consuming it?".

## When to use

Invoke this skill when:

- A named milestone (Milestone B, Milestone C, Plan 4, etc.) is ready to merge to `main`.
- The final commit for that milestone is on the feature branch.
- The merge-to-main PR is open (or about to be opened) but not yet merged.

**Trigger phrases that should call this skill:** "write release notes for Milestone X", "draft release notes", "/write-release-notes", "we're shipping Milestone X — do the notes", "release notes for the milestone PR".

Do NOT invoke for:

- Routine bug-fix branches that are not part of a named milestone.
- PRs without the `milestone` label.
- Post-merge cleanup (the notes belong in the merge PR itself, so reviewers can read them).

## Anti-patterns

- **Do NOT write release notes for non-milestone work.** A `fix:` PR that lands between milestones belongs in `CHANGELOG.md` only.
- **Do NOT just summarize raw commit messages.** Commit messages are the input; the notes are a re-told story for a reader who does not have the commit log open. Rephrase, group, contextualize.
- **Do NOT promise things that are not in the commits.** If a feature is half-shipped, mark it deferred. Do not write aspirational notes.
- **Do NOT skip the human-prose sections** ("Why it matters", "User-visible changes", "Operator-facing additions", "Deferred to next milestone"). A bullet list of file paths is not release notes.
- **Do NOT edit `CHANGELOG.md`.** Semantic-release regenerates it on every release tag. Any hand edits will be overwritten on the next release.
- **Do NOT push directly to main.** Release notes go on a `docs/<milestone>-release-notes` branch and merge via PR.
- **Do NOT invoke autonomously without a milestone signal.** Wait for an explicit user request or a clear milestone-PR context.

## The process

Follow these steps in order. Stop at each approval gate.

### 1. Confirm the milestone name

If the user did not supply a milestone name, ask: "What's the milestone name? (e.g. 'Milestone C', 'Plan 4 conversation deltas')". Get an answer before continuing. Derive the kebab-case slug from the name (e.g. "Milestone C" → `milestone-c`, "Plan 4 conversation deltas" → `plan-4-conversation-deltas`).

### 2. Auto-detect scope

Find the previous release-notes file and walk the git log forward from there:

```bash
# Latest existing release-notes file (or empty if none).
LATEST=$(ls -t docs/release-notes/*.md 2>/dev/null | grep -v _template.md | head -1 || true)

if [ -n "$LATEST" ]; then
  # Commit that introduced the previous notes file is the floor.
  LAST_SHA=$(git log -n1 --format=%H -- "$LATEST")
  git log --no-merges --format='%h %s' "$LAST_SHA"..HEAD
else
  # No prior notes — walk everything on the branch since main.
  git log --no-merges --format='%h %s' origin/main..HEAD
fi
```

Also pull PR context with `gh`:

```bash
# Open PRs labelled milestone — typically the merge PR for this milestone.
gh pr list --label milestone --state open --json number,title,body,headRefName

# Merged PRs since the last notes file (use the LAST_SHA date as the floor).
gh pr list --state merged --search "merged:>=$(git log -n1 --format=%cs $LAST_SHA)" \
  --json number,title,body,mergedAt
```

Group the commits and PRs into logical buckets that map to template sections (what shipped, fixes, deferred work, breaking changes). If a commit does not fit cleanly, that is a signal — either the commit message is bad or the milestone scope is fuzzy. Surface that to the user.

### 3. Read the template

```bash
cat docs/release-notes/_template.md
```

Treat the section headers as load-bearing. Add sections only if the milestone genuinely needs them (e.g. a new section for a deprecation), and call that out to the user.

### 4. Draft the file

Write to `docs/release-notes/YYYY-MM-DD-<kebab-milestone-name>.md` using today's date (`date +%Y-%m-%d`). Fill in every placeholder from the template. The narrative sections should be written for a reader who has not seen the commits — assume they are an operator deploying this version, or a mobile-client author wiring against the new API surface.

Use the existing `docs/release-notes/2026-06-04-milestone-b.md` as a tone reference. It is dense, specific, and operator-facing — match that register.

### 5. Self-review against the checklist

Before showing the draft to the user, verify each item:

- [ ] No placeholder strings remain (`TBD`, `TODO`, `<fill in>`, `<short milestone name>`, `XXX`).
- [ ] Every commit in the detected range is either represented in the notes or explicitly skipped (with a one-line reason). Do not silently omit commits.
- [ ] The "Why it matters" section is user-facing prose, not a restated commit list.
- [ ] Deferred-to-next-milestone items are called out explicitly with a "why deferred" note.
- [ ] Breaking changes are listed under "Breaking changes" — or the section explicitly says "None." Do not omit the section.
- [ ] The PR links and squash-commit SHA are filled in (use placeholder `<pending merge>` for the SHA if the merge has not happened yet).
- [ ] Test-suite count under "Testing" reflects the current `npm test` output, not a copy-pasted number from the previous milestone.

If any item fails, fix it before continuing.

### 6. Approval gate

Show the user:

1. The proposed filename.
2. The full diff (`git diff --no-index /dev/null docs/release-notes/<file>.md`).
3. The list of commits the notes cover.

Ask: "Approve and commit, or request changes?". Do not proceed without explicit approval.

### 7. Commit

After approval, check out a dedicated branch (if not already on one) and commit:

```bash
git checkout -b docs/<milestone-slug>-release-notes
git add docs/release-notes/<file>.md
git commit -m "docs(release-notes): add <milestone-name> notes"
```

Title format follows the global conventional-commit rule (lowercase, no period, imperative). Do not add a `Co-Authored-By` trailer.

### 8. Push and open draft PR

```bash
git push -u origin docs/<milestone-slug>-release-notes
gh pr create --draft --title "docs(release-notes): <milestone-name>" \
  --body "Release notes for <milestone-name>. Companion to the milestone merge PR."
```

The PR can be marked ready for review (or squash-merged) by the user once the milestone PR itself merges.

## What NOT to do

- Do NOT invoke this skill autonomously — wait for a milestone signal.
- Do NOT push directly to `main`.
- Do NOT skip the approval gate in step 6.
- Do NOT modify `CHANGELOG.md` under any circumstances.
- Do NOT add features, abstractions, or sections to the template "while you're in there". The template is the contract.
- Do NOT use `--no-verify` to bypass the commit hooks if the title is rejected — fix the title.
